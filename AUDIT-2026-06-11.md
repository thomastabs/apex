## Executive Summary

Codebase is in good shape: thin routers, clean service layer, real adapter abstraction,
meaningful prompt engineering. But the audit found **one shipped regression that breaks
Phase 4**, **two systemic security gaps** (no real authentication; SSRF bypass via the
config path), and a **re-introduced NaN bug** in the Taiga adapter context. None are
caught by the current test suite тАФ Phase 4 has zero backend tests, and the latest
refactor (`6147855`) deleted functions that production code still calls.

## 2. Security & Authentication

> Assumes backend is internet-reachable (Azure Container Apps public ingress per ci.yml).

### ЁЯФ┤ C2 тАФ No actual authentication on the backend
- [x] **FIXED 2026-06-11 (Option A, commit f4e32a1)** тАФ `get_auth_context` validates the token against the anchored PM (`Taiga /users/me` / Jira `/myself`); `get_request_context` authorizes `X-Project-Id` via project readability on the PM (403 cross-tenant). Identity provider resolved server-side (TAIGA_API_URL env / config), never from headers. 60s positive / 10s negative cache, bounded 10k entries. Per-IP cap (30/min) added to AI rate limiter.
- **Deployment note:** self-hosted Taiga must set `TAIGA_API_URL` so tokens validate against the right instance. Anchor resolution (2026-06-12): `TAIGA_API_URL` env тЖТ config `taiga_url` тЖТ Taiga Cloud, all SSRF-validated. **Verified live 2026-06-12** against a local taiga-docker instance via Cloudflare tunnel: proxy login, real-token 200 / bogus-token 401, own-project 200 / foreign-project 403, phase endpoint end-to-end. README private-Taiga guide updated with the backend `TAIGA_API_URL` step.
- Residual (accepted, ties to H4): workspace config is global тАФ any PM-authenticated user can flip `pm_tool`/`jira_base_url`, breaking the identity anchor for others (DoS-class). Real fix = per-user/project config scoping.

### ЁЯФ┤ C3 тАФ SSRF: config-path URLs bypass all validation
- [x] **FIXED 2026-06-11** тАФ shared `backend/app/api/ssrf.py`; `_get_taiga_url` config branch and `_get_jira_base_url` now run the same validators as the header paths; `save_config` validates `jira_base_url` at write time (empty string still clears).
- Was: `POST /workspace/config {"pm_tool":"jira","jira_base_url":"http://169.254.169.254"}` тЖТ any `/api/pm/jira/*` call hits cloud metadata service.

### ЁЯЯа H3 тАФ DNS-rebinding hole in SSRF guard
- [x] **FIXED 2026-06-11** тАФ `ssrf.is_blocked_host` resolves hostnames via `getaddrinfo` and blocks any private/loopback result; unresolvable names pass through (connection fails anyway). Residual TOCTOU documented: full fix needs resolved-IP pinning in httpx.

### ЁЯЯб M5 тАФ Open auth-relay on `/api/pm/taiga/auth`
- [ ] Unauthenticated, un-rate-limited, forwards credentials to any public https host's `/api/v1/auth` тЖТ credential-stuffing relay hiding attacker IP. Add IP rate limit; consider host allowlist.

### ЁЯЯб M6 тАФ Prompt injection surface
- [ ] User text (epic descriptions, hints, QA notes) and **`github-context.md`** (README from an EXTERNAL repo) injected into prompts undelimited. Hostile README тЖТ injected into Phase 2/3 generation тЖТ persisted into specs feeding later phases (cross-phase chain).
- Fix: fence untrusted segments with "data, not instructions" preamble; cap github-context length; keep DOMPurify invariant on all markdown render surfaces.

### ЁЯЯб M7 тАФ XSS blast radius: tokens in sessionStorage + permissive CSP
- [ ] CSP has `'unsafe-inline' 'unsafe-eval'`, `connect-src тАж https:` (any host). Single XSS exfiltrates PM tokens anywhere. Long-term: nonce-based CSP via middleware; tighten connect-src to API + GitHub.

### ЁЯЯб M8 тАФ Proxy responses unbounded
- [ ] 4 MB cap is request-only; `resp.content` from Taiga/Jira buffered fully with no limit. Add response size cap (~10 MB) or stream.

**Done well:** SSRF netblock list for literals; `\r\n` header-injection guards; token length caps; PAT excluded from persistence; `save_config` pops `auth_token`; `follow_redirects=False` on proxy clients.

## 3. Architecture & SDLC Alignment

### ЁЯЯа H4 тАФ Workspace config is a global singleton (accidental single-tenancy)
- [ ] `.apex-config.json` holds `pm_tool`/`jira_base_url`/`project_id`/`ai_model` for the entire backend. `set_active_project` (`context_manager.py:158-168`) WRITES `project_id` on every project switch тАФ write in a read path, thrashes under concurrent users; user A switching to Jira breaks user B's Taiga config-path proxy.
- Fix: scope config per project dir (or per user); remove the `save_config` side effect from `set_active_project`.

### ЁЯЯб M9 тАФ Phase boundary leaks (otherwise clean)
- [ ] `phase4_service` imports `context_manager` directly (lines 101, 139, 152) instead of `ContextService` тАФ only service that does.
- [ ] `workspace.py:86`: `ai_engine._llm_cache.clear()` тАФ router touching a private; clears cache during in-flight AI calls on other threads.

### ЁЯЯб M10 тАФ AI engine blocking retries multiply
- [ ] `_ai_retry` (10s + 30s sleeps) wraps clients already at `max_retries=2`, on sync endpoints on the ~40-thread anyio pool. Burst of 429s can starve ALL sync endpoints.
- Fix: async AI endpoints with `asyncio.sleep`, or cap combined retry budget, or dedicated smaller threadpool.

### ЁЯФ╡ L3 тАФ `get_model()` re-reads config from storage per AI call (network round trip in Azure mode). Cache with short TTL.
### ЁЯФ╡ L4 тАФ `context_manager.__getattr__` magic masks deletions (why C1 surfaced at request time, not import time). Plan removal; tests are the only consumer.
### ЁЯФ╡ L5 тАФ `PHASE_STATUSES` lacks `new` status that CLAUDE.md documents. Align.

## 6. Phase 5 тАФ Research & Planning

`ai_engine.py:1460-1467` TODO + placeholder page frame Phase 5 as **Deployment**.

### Feature A тАФ Release Readiness & Deployment Gate (natural Phase 5)
- Release manifest from `qa_passed` stories; AI release notes from Gherkin + bug reports.
- `generate_infra_delta` (existing TODO): DevOps persona scans tech spec тЖТ `INFRA_DELTA: NONE | REQUIRED <delta>`.
- Deploy checklist + human gate тЖТ `mark_story_deployed` (already implemented) тЖТ PM status transition.
- Effort: low тАФ reuses every existing pattern (1 prompt, ~3 endpoints, 1 workflow component).

### Feature B тАФ Code Verification Loop (highest thesis value)
- Fetch merged PR diff via existing GitHub client; AI-as-judge: diff + Gherkin + pack test assertions тЖТ **traceability matrix** (scenario тЖТ evidence тЖТ covered/uncovered/contradicted).
- Closes the feedback loop: Apex currently generates instructions but never verifies results. Pre-seeds Phase 4 QA with "scenarios the diff doesn't cover."

### Feature C тАФ SDLC Analytics & Retrospective (evaluation instrument)
- Add timestamps to story-index transitions; dashboard: cycle time per phase, Fix-Bolt rate per epic, vaccine recurrence, AI token/cost per story (log in `_invoke`).
- AI sprint retrospective grounded in the data. Gives quantitative thesis evaluation data.

**Recommendation:** A + B for Phase 5; C alongside as the evaluation instrument. Skip full agentic autonomy (auto-executing packs) тАФ thesis-sized scope expansion; the human gate IS the framework's point.
